# 技能：功能啟動與分析計畫 (Feature Kickoff & Analysis Planning)

**目的**：確保每一個功能的開發都始於經驗證的事實、明確的計畫以及使用者的明確核准 —— 在觸及任何實作檔案之前。

> **專案特定細節**（建置指令、驗證指令、命名慣例）來自於 `copilot-instructions.md`，該檔案始終存在於上下文（Context）中。

---

## 執行觸發

當使用者執行以下操作時，自動啟用此技能：
- 請求新功能或增強功能（"add X", "implement Y", "we need Z"）
- 提供要實作的需求或規格
- 要求跨越多個檔案的重構 (Refactor)
- 說出類似 "start working on feature N" 的話

---

## 執行步驟

### 步驟 1 — 解析請求
從使用者的訊息中擷取：
- **功能名稱 / 代號 (Slug)**（用於資料夾命名，例如 `user-session-metrics`）
- **受影響區域**（涉及哪些頁面、控制器、API 或層級）
- **規模信號**（小調整 vs. 多層次變動）

---

### 步驟 2 — 探索背景 (Context Discovery)
搜尋程式碼庫以了解目前的狀態。使用專案適用的搜尋指令：

- 尋找受影響的原始碼檔案（頁面、組件、控制器）
- 尋找 API 路由 (Route) 定義
- 尋找服務介面 (Service interfaces) 與資料合約 (Data contracts)
- 尋找相關的資料模型 (Data models) / DTOs

> **請參閱 `copilot-instructions.md`** 以獲得專案特定的搜尋指令與目錄慣例。

盡可能地並行執行搜尋。

---

### 步驟 3 — 代碼驗證清單 (強制執行)

**在呈現計畫之前，每一項事實陳述都必須經過原始碼驗證。**
針對每一項聲稱的事實，記錄原始檔案路徑與行號。

| 事實類型 | 如何驗證 | 必要證據 |
|---|---|---|
| 檔案存在 | 搜尋專案中的檔案名稱 | 確認絕對路徑 |
| API 路由 | 在 controller/router 中 grep 路由屬性 | `檔案名稱:行號` |
| 頁面/組件調用端點 | 在原始碼中 grep API URL 模式 | `檔案名稱:行號` |
| 組件名稱 | 閱讀原始碼中的常數或配置 | 代碼中的精確字串 |
| 資料模型與屬性 | 閱讀實際的 class/interface 定義 | 類別名稱 + 關鍵屬性 |

**警示符號 (Red Flags) — 在繼續之前請停止並驗證：**
- ❌ 文件說一套，代碼顯示另一套
- ❌ 任何沒有代碼行號支持的假設
- ❌ 名稱是經過翻譯或猜測的，而不是從原始碼中讀取的
- ❌ 從文件中複製路由字串，而未在代碼中確認

---

### 步驟 4 — 決定功能序號
```bash
ls spec/features/ | sort
```
取得下一個可用的 3 位數編號（例如，若 `004-security-fix` 已存在，則下一個是 `005`）。

---

### 步驟 5 — 建立功能資料夾鷹架 (Scaffold)
建立功能資料夾，並填入範本內容：

```
spec/features/NNN-{slug}/
├── README.md                  ← 來自 feature-readme-template.md
├── FEAT-NNN-requirements.md   ← 來自 requirements-template.md
└── PHASE-{N}-PROGRESS.md      ← 當實作開始時建立 (來自 phase-progress-template.md)
```

> **注意**：`PHASE-{N}-PROGRESS.md` 在啟動階段「不」建立 —— 它是該階段實作開始時才建立。每個階段一個檔案。

根據步驟 1–3 所知的內容填寫。針對尚待決定的事項保留佔位符。

建立檔案後，**在 `README.md` 中設定初始狀態**：
- 設定 `Status` = 🟡 Planning
- 設定 `Updated` = 今天日期

---

### 步驟 6 — 起草實作計畫
將計畫結構化如下：

```markdown
## 已尋獲背景 (Context Found)
- [已發現的檔案與路徑]
- [目前使用的模式 (附帶 檔案:行號 引用)]
- [已確認的目前 API 路由與資料模型]

## 建議實作 (Proposed Implementation)
- [欲修改的檔案 — 及其理據]
- [欲建立的檔案 — 及其理據]
- [欲遵循的方法與架構模式]

## 階段分析 (若 FR > 3 或依賴圖有分支)
- [FR 依賴圖 — 哪些 FR 依賴於哪些]
- [建議的階段切分 — 哪些 FR 歸類在每個階段]
- [針對每個階段：目標、向後相容性、回滾計畫]
- [若 FR ≤ 3 且為線性：註明「單一階段 — 直接套用深度規則」]

## Schema / 資料模型變更 (若適用)
- [需要的資料庫 / Schema 變更]
- [考慮過的替代方案]
- [遷移策略]

## 需要釐清的問題 / 決策
- [供使用者選擇的選項]
- [編碼前需要解決的權衡方案]

## 驗證計畫
- [專案的建置指令]
- [功能測試步驟]
- [預期輸出]
```

---

### 步驟 7 — 呈現計畫並等待核准

向使用者呈現計畫。**在使用者明確核准前，不得編輯任何實作檔案。**

清楚說明：
- 計畫的範圍內 (In Scope) 包含什麼
- 範圍外 (Not in scope) 包含什麼（延後至後續階段）
- 任何在編碼開始前需要回答的開放性問題

---

## 規則

- ❌ **絕不**在核准前編輯實作檔案
- ❌ **絕不**在沒有代碼證據的情況下陳述事實（檔案路徑、路由、組件名稱、資料模型）
- ❌ **絕不**猜測或翻譯名稱 —— 始終從原始碼讀取
- ✅ 將建立的功能資料夾路徑作為計畫輸出的一部分
- ✅ 若使用者提供 `[JUST DO IT]` 或 `[SKIP CLARIFY]`，則直接進入實作，但仍須靜默執行驗證檢查

---

## 輸出檢查清單 (在呈現計畫前)

- [ ] 已建立功能資料夾 `spec/features/NNN-{slug}/`
- [ ] `README.md` 的 Status = 🟡 Planning, Updated = 今天
- [ ] 引用的每個檔案路徑皆存在（已透過檔案搜尋確認）
- [ ] 引用的每個 API 路由皆與實際代碼屬性相符
- [ ] 每個組件/頁面名稱均來自原始碼，非猜測
- [ ] 計畫中至少包含一個驗證指令
- [ ] 已註明範圍邊界（現在不實作什麼）
