# 技能：需求調整 (Requirements Realignment)

**目的**：當實作過程顯示需求或計畫需要調整時，確保在**繼續**編碼之前更新所有規劃文件。防止過時的文件導致下游混淆、遺漏範圍或錯誤實作。

> **專案特定細節**（規格目錄、功能資料夾慣例、建置指令）來自於 `copilot-instructions.md`，該檔案始終存在於上下文（Context）中。

---

## 執行觸發

當發生以下情況時啟用此技能：
- 實作過程顯示設計假設錯誤或不完整
- 使用者在實作過程中選擇了新方法 / 替代解決方案
- 需要原本需求中沒有的新功能需求 (FR)
- 現有 FR 的範圍發生實質變更（新的驗收標準、不同的行為）
- 做出變更計畫的架構決策 (AD/Q)
- 使用者說出「更新需求」、「計畫變更了」、「我們需要調整範圍」
- 代理人 (Agent) 偵測到代碼變更與記錄的計畫不再匹配

---

## 核心原則

```
實作揭示問題 → 討論選項 → 使用者選擇方法
    → 1) 更新需求文件 (要建置「什麼」)
    → 2) 更新進度/計畫文件 (我們如何「追蹤」它)
    → 3) 恢復實作
```

**在恢復實作之前，必須完成 (1) 和 (2)。** 僅更新其中一個會導致資訊不對稱 —— 需求文件說一套，而進度追蹤器卻顯示另一套。

---

## 執行步驟

### 步驟 1 — 識別變更內容

簡明扼要地總結差異 (Delta)：

| 問題 | 回答 |
|---|---|
| 什麼原因觸發了變更？ | (例如：「執行階段測試顯示 schema topic 僅包含已註冊的資料表」) |
| 舊有的假設是什麼？ | (例如：「schema topic 僅需要具備活動處理程式的資料表」) |
| 新的決策是什麼？ | (例如：「schema topic 必須涵蓋所有 MySQL 資料表，以供未來的處理程式接入」) |
| 哪些 FR 受到影響？ | (例如：「新增了 FR-9；FR-6 的負載格式已變更」) |
| 哪些文件需要更新？ | (始終包含：需求文件 + 進度文件；選填：README, BACKLOG) |

---

### 步驟 2 — 更新需求文件

開啟功能的需求文件（例如 `FEAT-NNN-requirements.md`）並套用變更：

| 變更類型 | 更新位置 |
|---|---|
| 新的 FR | 新增 FR 區塊，包含完整的 AC (驗收標準)、優先級、描述 |
| 修改後的 FR | 更新現有 FR 的 AC、描述或範圍 |
| 新決策 (Q/AD) | 在「決策」表格中新增一行並附帶理據 |
| 架構 / 資料流變更 | 更新圖表與流程描述 |
| 階段映射變更 | 更新「FR → 階段映射」表格 |
| 負載 (Payload) / Schema 變更 | 更新訊息格式 / 資料模型區塊 |

**更新文件標頭中的 `Updated` 日期**。

---

### 步驟 3 — 更新進度 / 計畫文件

開啟階段進度文件（例如 `PHASE-N-PROGRESS.md`）並套用變更：

| 變更類型 | 更新位置 |
|---|---|
| 範圍內的新 FR | 新增至 FR 範圍表 (狀態為 ⬜) |
| 需要新步驟 | 插入步驟（例如步驟 3a）並註明計畫範圍 |
| 現有步驟範圍變更 | 更新該步驟的計畫或已完成描述 |
| 觀測能力 / 指標變更 | 更新步驟 4 的計畫範圍 |
| 回滾計畫受到影響 | 更新「回滾計畫」區塊 |

**更新文件標頭中的 `Updated` 日期**。

---

### 步驟 4 — 新增需求調整日誌項目 (Realignment Log)

附加至進度文件中的「需求調整日誌」區塊。若該區塊不存在則建立它。

格式：

```markdown
## 需求調整日誌 (Realignment Log)

| 日期 | 觸發原因 | 變更內容 |
|---|---|---|
| YYYY-MM-DD | [導致變更的原因] | [已完成的文件更新總結] |
```

這會建立一個關於計畫演進原因的審核軌跡，對於事後回顧與新進人員加入非常有價值。

---

### 步驟 5 — 向使用者呈現總結

在恢復實作之前，呈現一份簡潔的差異總結：

```markdown
**需求調整完成。** 已更新：
- 需求文件：[變更清單]
- 進度文件：[變更清單]
- 需求調整日誌：[已新增項目]

已準備好繼續進行 [下一個步驟/FR]。
```

---

## 規則

- ❌ **絕不**在範圍/設計變更後，未先更新文件就繼續實作
- ❌ **絕不**僅更新需求文件 —— 進度文件也必須同步
- ❌ **絕不**僅更新進度文件 —— 需求文件是「什麼 (WHAT)」的唯一事實來源
- ✅ 兩份文件皆已更新 = 可以安全恢復編碼
- ✅ 需求調整日誌項目 = 保留了審核軌跡
- ✅ 若使用者說出 `[SKIP DOCS]`，請在註釋中明確記錄此技術債，但仍須建立需求調整日誌項目

---

## 反面模式 (Anti-Patterns)

| 反面模式 | 為什麼不好 | 正確做法 |
|---|---|---|
| 「我晚點再更新文件」 | 文件產生偏離；下次對話時背景已過時 | 現在更新，在下一次代碼編輯之前 |
| 更新需求但未更新進度 | 進度追蹤器顯示錯誤的範圍/步驟 | 始終同時更新兩者 |
| 更新進度但未更新需求 | 需求文件對於新讀者來說已過時 | 始終同時更新兩者 |
| 沒有需求調整日誌 | 沒有關於計畫為何改變的審核軌跡 | 始終新增日誌項目 |
| 先實作，後記錄 | 冒著基於過時假設進行開發的風險 | 文件優先，代碼隨後 |

---

## 輸出檢查清單

在需求調整後恢復實作前：

- [ ] 需求文件已更新，包含新/變更的 FR、決策或架構
- [ ] 需求文件的 `Updated` 日期為今天
- [ ] 進度文件已更新，包含新/變更的 FR 範圍、步驟或回滾計畫
- [ ] 進度文件的 `Updated` 日期為今天
- [ ] 已新增需求調整日誌項目，包含日期、觸發原因與變更內容
- [ ] 已向使用者呈現差異總結
- [ ] 已準備好恢復實作
