# AI 輔助開發工作流程

**人機協作開發的基礎原則。通用且可重用 — 可複製到任何專案。**

> **落實方式**：這些原則由 `skills/` 中的 Copilot Skills 實作：
> - 功能啟動與需求確認 → `skills/feature-kickoff/`
> - 建置驗證與完成確認 → `skills/implementation-completion/`
> - 錯誤調查與診斷 → `skills/debug-investigation/`
> - 進度追蹤與文件同步 → `skills/status/`
>
> `.github/agents/` 中的 **Agents** 提供專屬模式（例如：`High-Level-Big-Picture-Architect` 用於在實作開始前進行架構分析與圖表生成）。

---

## 核心原則

### 1. 先釐清再撰寫程式碼（預設工作流程）
- **在編輯檔案前，務必先呈現實作計畫**
- 先蒐集背景資訊：搜尋程式碼、閱讀 schema、理解現有模式
- 呈現內容：發現了什麼、建議的變更、替代方案、待確認問題
- 等待**明確核准**後再編輯檔案
- 例外：簡單且低風險的變更（讀取程式碼、明顯的 bug 修正、註解/日誌）
- **Agents 可加速此流程**：在任何程式碼規劃開始前，使用 `High-Level-Big-Picture-Architect` 進行架構分析、圖表繪製及潛在失敗模式識別

### 2. 漸進式交付，而非一次完整解決方案
- 優先交付**最小可驗證單元**
- 每次迭代必須**可獨立測試**
- 在核心功能驗證完成前，暫緩加入生產環境模式

### 3. 明確優於隱含
- **絕不假設背景資訊存在** — 主動詢問缺少的部分
- 清楚說明每個階段**不會**實作的內容
- 明確宣告系統邊界

### 4. 驗證優先
- 程式碼必須**可立即執行/測試**
- 提供**精確的驗證指令**
- 提供**可實際使用的測試資料範例**

### 5. 區分「現在」與「之後」
- 現在實作**最小可運作版本**
- 將**延後工作**記錄在 spec 文件中，標記為 Phase ⬜ 項目或 ⬜ 狀態的 FR
- 保持未來階段的可見性，但在當前階段驗證完成前不予實作

### 6. Schema / 資料模型變更需設置檢查點
- 在實作資料庫 schema 變更前，**停下來並呈現提案**
- 提案中包含 SQL/schema 定義與替代方案
- 等待**明確核准**後再產生程式碼
- Schema 變更代價高昂 — 務必一次做對

## 階段規劃 — 將功能切割為可交付單元

> **為何重要**：下方的「深度規則」說明在單一模組中要做到多深。但這些規則假設你已知道*哪些 FR 應一起處理*。當一個功能有許多 FR 且依賴關係圖有分支時，同時實作所有內容會讓驗證變得不可能，且回滾風險極高。

### 何時需要切割

| 情境 | 行動 |
|---|---|
| FR ≤ 3，線性依賴鏈 | 跳過階段規劃 — 直接套用深度規則 |
| FR > 3 **或** 依賴關係圖有分支 | 分析 FR 依賴關係 → 切割為**工程階段** |

### 如何切割

1. **繪製 FR 依賴關係圖** — 哪些 FR 依賴哪些 FR。
2. **識別自然切割點** — 可獨立部署的 FR 群組。
3. **每個階段必須**：
   - **可獨立部署** — 僅完成此階段後，系統仍可運作（可能功能有所縮減）。
   - **向下相容** — 不破壞既有行為。若會改變既有行為，須以 Feature Flag 控管。
   - **可獨立驗證** — 具備自己的驗收標準與測試步驟。
4. **每個階段需記錄**：目標、FR 對應、向下相容聲明、回滾計畫。
5. **在每個階段內套用深度規則（4 個步驟）**，而非一次跨所有 FR 進行。

### 在哪裡記錄

- **`FEAT-NNN-requirements.md`** — 「實作階段」章節，包含 FR → 階段對應表及各階段詳細說明。
- **`README.md`** — 標頭表格中的 Phase 欄位追蹤目前進行中的階段。
- **`BACKLOG.md`** — 任何延後的階段在此新增一列。
- **`PHASE-{N}-PROGRESS.md`** — 當一個階段的實作開始時建立。追蹤每步驟的進度（建立/修改的檔案、建置/測試結果、決策）。每步驟通過建置與測試後更新。跨 session 保存進度，確保進度不遺失。範本：`spec/_templates/FEATURES/phase-progress-template.md`。

### 範例

```
FR-1 ──┐
FR-2 ──┼──→ Phase 1：DDL Pipeline（可獨立部署，不改變資料路徑）
FR-3 ──┤
FR-6 ──┘

FR-4 ──┐
FR-5 ──┼──→ Phase 2：動態資料路徑（依賴 Phase 1 的 schema cache）
FR-7 ──┤
FR-8 ──┘
```

Phase 1 上線並在 SIT 驗證。Phase 2 待 Phase 1 穩定後才開始。

## 實作深度 — 在每個階段內的漸進式交付

> **為何重要**：若沒有此約束，AI 會在單次輸出中產生橫跨多個檔案的生產級程式碼 — 重試佇列、熔斷器、指標、完整錯誤處理 — 使得連基本功能是否正常都無法驗證。

### 原則：每步驟交付最小可驗證單元

在任何功能階段內（例如：「Phase 2：Debezium Kafka Signal Producer」），實作依照以下進程：

### Step 1 前 — 建立階段進度文件
開始某一階段的實作時，從 `spec/_templates/FEATURES/phase-progress-template.md` 建立 `spec/features/NNN-{slug}/PHASE-{N}-PROGRESS.md`。此文件在每步驟通過建置與測試驗證後更新。

### Step 1 — 核心行為（可觀察）
- 實作**單一核心動作**（例如：產生一則 Kafka 訊息）
- 必須**可獨立驗證**（例如：在 Kafka topic 中看到該訊息）
- ✅ 在專用設定類別中使用硬編碼值是可接受的
- ✅ 結構化日誌以確認行為
- ❌ 錯誤處理僅限於 log-and-continue
- ❌ 尚未與上下游整合

### Step 2 — 整合（已連接）
- 接入實際呼叫點（例如：注入至 handler 失敗路徑）
- 驗證**端對端**觸發是否正常（例如：merge 失敗 → 產生 signal）
- ✅ 從設定檔讀取真實設定
- ✅ 完成 DI 註冊
- ❌ 尚未處理邊緣案例的防禦性程式碼

### Step 3 — 彈性（強化）
- 為整合邊界加入錯誤處理
- 優雅地處理失敗（例如：signal 產生失敗 → 記錄日誌，不讓系統崩潰）
- ✅ 在系統邊界進行驗證
- ✅ 外部依賴失敗時的回退機制

### Step 4 — 可觀測性（監控）
- Prometheus 計數器、NLog 警報、Dashboard 設定
- 僅在核心 + 整合 + 彈性皆驗證完成後才加入

> **規則**：每個步驟必須**核准並驗證**後才能進行下一步。絕不合併步驟。
>
> **規則**：每個步驟通過建置與測試後，在 `PHASE-{N}-PROGRESS.md` 中更新建立/修改的檔案、驗證結果及決策記錄。此為強制要求 — 確保進度跨 session 不遺失。

## 錯誤處理深度

錯誤處理同樣遵循漸進式深度原則 — 從最小化開始，驗證核心行為後再逐層加入：

| 步驟 | 錯誤處理 | 範例 |
|------|---------|------|
| Step 1（可觀察） | `try/catch → log → continue` | Signal 產生失敗 → 記錄錯誤，return |
| Step 2（已連接） | + 傳播決策 | Signal 失敗 → handler 是否仍應回傳 `true`？ |
| Step 3（強化） | + 型別化例外、回退機制 | Kafka 無法連線 → 回退為僅記錄日誌模式 |
| Step 4（監控） | + 指標、警報 | `bqproxy_cdc_signal_error_total` 計數器 |

> **專案特定**的錯誤處理模式（middleware、exception 類型），請參考 `copilot-instructions.md`。

## 延後工作追蹤

計畫中存在但目前不實作的工作，記錄在 **spec 文件**中，而非程式碼註解：

| 機制 | 位置 | 範例 |
|------|------|------|
| Phase ⬜ 狀態 | `FEAT-NNN-requirements.md` | `### Phase 3: Monitoring & Alerting ⬜` |
| FR/NFR ⬜ 狀態 | 需求表格 | `FR-08 \| Prometheus counter \| Should \| ⬜` |
| TBD ❓ | 問題章節 | `TBD-5 ❓: Delete events — 處理還是忽略？` |
| AD ❓ | 架構決策 | `AD-6: Delete Events ❓` |

避免在程式碼中使用 `// TODO-PRODUCTION:` 註解 — 這類註解會過時且被忽略。Spec 文件才是延後工作的唯一事實來源。

## 應遵循的常用模式

### 務必使用這些模式
1. **依賴注入（Dependency Injection）**：所有依賴透過建構子注入
2. **Async/Await**：所有 I/O 操作均為非同步
3. **結構化日誌（Structured Logging）**：日誌語句使用參數佔位符
4. **設定檔（Configuration）**：使用設定檔，不硬編碼
5. **單一職責（Single Responsibility）**：一個類別 = 一個變更原因

> **專案特定**的模式（命名、縮排、日誌、global usings），請參考 `copilot-instructions.md`。

### 在正確步驟前暫緩這些事項
1. **重試邏輯（Retry Logic）**：Step 3（彈性）— 核心 + 整合驗證後
2. **熔斷器（Circuit Breakers）**：僅在觀察到真實可用性問題時才加入
3. **完整驗證（Comprehensive Validation）**：Step 3 — Step 2 先從 null 檢查開始
4. **單元測試（Unit Tests）**：在實作穩定後才撰寫
5. **效能優化（Performance Optimization）**：先讓功能可運作，再優化

## 快速參考

### 任何實作開始前
1. ✅ 架構分析（HLBPA agent 或同等工具）— 圖表、失敗模式、TBDs
2. ✅ 功能啟動 — spec 文件建立完成，需求已包含階段規劃
3. ✅ Spec 核准 — 目前階段的 AD 已決定、TBD 已解決
4. ✅ 接著**在階段內一次一步**進行實作

### 可讀性檢查清單（務必驗證）：
- [ ] 每個元件都有一個明確的職責
- [ ] 介面聚焦（無臃腫介面）
- [ ] 主要協調者與各實作分離
- [ ] 日誌使用結構化參數
- [ ] 所有依賴皆已注入
- [ ] 沒有元件超過約 250 行
- [ ] 命名清晰且具自我說明性

### 應避免的事項：
- ❌ 假設背景資訊存在 — 先搜尋程式碼
- ❌ 建立龐大的上帝類別（900 行的類別）
- ❌ 混合業務邏輯與持久化邏輯
- ❌ 硬編碼設定值
- ❌ 隱式依賴（緊耦合）
- ❌ 在單次交付中實作多個步驟
- ❌ 過早加入抽象層
- ❌ 實作不在當前階段範圍內的模式
- ❌ 撰寫程式碼時未包含驗證步驟

### 記住：
> **最小可驗證單元** > 完整解決方案
> **Spec 文件** > 程式碼註解（用於追蹤延後工作）
> **可讀、聚焦的元件** > 龐大的便利性設計

## 單體 ❌ vs. 聚焦元件 ✅ 範例

### ❌ 反模式 - 單體實作（900 行）
```
// 問題：單一類別負責所有事
class DataProcessor
    - 從 API 抓取資料（200 行）
    - 轉換與豐富資料（200 行）
    - 偵測異常（200 行）
    - 建立事件（150 行）
    - 發送通知（150 行）

問題：
- 難以獨立測試各個關注點
- 難以在其他地方重用轉換邏輯
- 難以理解（900 行！）
- 一處變更會影響多個關注點
- 違反單一職責原則
```

### ✅ 模式 - 聚焦元件
```
// 介面（整合、組織化）
interface DataFetcher: fetch(source) → data
interface DataTransformer: transform(data) → enriched_data
interface AnomalyDetector: detect(data) → boolean
interface IncidentCreator: create(data) → incident
interface NotificationSender: send(incident) → void

// 實作（每個約 100-150 行）
class APIDataFetcher(config): DataFetcher
  - 僅負責從 API 抓取
  - 處理 API 重試/錯誤
  
class DataTransformer(rules): DataTransformer
  - 僅負責資料轉換
  - 純邏輯，無副作用
  
class StatisticalAnomalyDetector(threshold): AnomalyDetector
  - 僅負責偵測異常
  - 回傳布林值
  
class IncidentCreator(repo): IncidentCreator
  - 僅負責持久化事件
  - 資料庫操作
  
// 協調者（清晰的資料流，約 80 行）
class DataProcessingOrchestrator
    def process(source):
        data = fetcher.fetch(source)              // Step 1: 抓取
        enriched = transformer.transform(data)    // Step 2: 轉換
        if detector.detect(enriched):             // Step 3: 偵測
            incident = creator.create(enriched)   // Step 4: 建立
            sender.send(incident)                 // Step 5: 通知
```

**優點：**
- ✅ 每個元件可獨立測試
- ✅ 易於理解各職責
- ✅ 元件可重用
- ✅ 一個關注點的變更不影響其他關注點
- ✅ 協調者中可見清晰的資料流
- ✅ 高度可讀且易維護
- ✅ 遵守單一職責原則
- ✅ 支援依賴反轉原則

## 寫程式前先讀程式碼（強制）

**關鍵實踐**：在實作應遵循現有模式的新程式碼時，務必先檢查 codebase 中的既有範例。絕不假設：
- 資料結構路徑（例如：`data.property` vs `data.summary.property`）
- 類似功能的命名慣例
- 回應物件層級結構
- 變數範疇與賦值模式

**為何重要**：答案幾乎總是已存在於 codebase 中 — 你只需要先找到既有的可運作程式碼。

**調查模式**：
1. 搜尋使用相同資料來源或類似功能的既有程式碼
2. Grep 相關欄位名稱以找到可運作的存取模式
3. 在完整背景下閱讀可運作的程式碼
4. 將完全相同的模式套用至新程式碼

> 最便宜的 bug 是從不引入的 bug。先讀程式碼可防止基於假設的錯誤。

## 需求重新對齊 — 處理實作中途的變更

> **實作方式**：`skills/requirements-realignment/`

實作過程中常常會發現設計假設有誤、範圍需要調整，或出現新的需求。當此情況發生時，**必須在繼續撰寫程式碼前更新所有規劃文件**。

### 模式

```
實作揭露問題 → 討論選項 → 使用者選擇方案
    → 1) 更新需求文件（建置「什麼」）
    → 2) 更新進度/計畫文件（「如何」追蹤）
    → 3) 新增重新對齊日誌條目（「為何」計畫改變）
    → 4) 繼續實作
```

### 何時觸發

- 出現原始需求中未包含的新 FR
- 使用者選擇了與原始計畫不同的方案
- 架構決策改變了現有 FR 的範圍
- 實作測試發現了設計缺口

### 規則

- 需求文件與進度文件**兩者**都必須更新 — 只更新其一會造成資訊分裂
- `PHASE-N-PROGRESS.md` 中的**重新對齊日誌**保存稽核軌跡
- 絕不在過時的計畫上繼續撰寫程式碼 — 5 分鐘的文件更新遠比實作錯誤的東西代價低

### 重新對齊日誌格式

```markdown
| 日期 | 觸發原因 | 變更內容 |
|---|---|---|
| YYYY-MM-DD | [造成變更的原因] | [文件更新摘要] |
```

## 交付前建置驗證（強制）

**關鍵**：在編譯/建置成功前，程式碼**尚未完成**。在宣告任務完成前：

1. 對受影響的專案執行建置指令
2. 驗證 **0 個編譯錯誤**（警告可接受）
3. 確認建置成功完成

**驗證檢查清單**：
- ✅ 對受影響的專案執行建置
- ✅ 驗證 **0 個錯誤**（警告可接受）
- ✅ 確認建置成功
- ❌ 建置失敗時絕不宣稱程式碼已完成
- ❌ 未驗證編譯前絕不告訴使用者「程式碼已就緒」

**交付狀態參考**：

| 建置狀態 | 行動 |
|---|---|
| ✅ 0 個錯誤 | 建立完成報告 → 交付 |
| ⚠️ 0 個錯誤，N 個警告 | 交付，並備註警告 |
| 🔴 N 個錯誤 | 先修正 → 重新建置 → 不予交付 |

**良好的完成訊息**：
```
✅ 程式碼實作完成。已驗證：
- 建置成功 — 0 個錯誤，N 個警告
- 修改了 [N] 個檔案
- 已可進行程式碼審查與測試
```

**不良的完成訊息**（絕不這樣做）：
```
✗ 我已完成變更的實作。               ← 未進行建置驗證
✗ 程式碼應該可以正確編譯。            ← 假設，非證據
✗ 已準備好測試（邏輯看起來對我是正確的）。← 未執行建置 = 尚未就緒
```

## 文件準確性規則（強制）

**原則**：文件必須反映現實。不準確的文件比沒有文件更糟。

### 依據程式碼驗證，而非依靠記憶
- ❌ **絕不**憑空捏造或猜測名稱、路由或類別名稱
- ✅ **務必**閱讀實際原始碼以確認事實
- ✅ **務必**為每個事實性主張引用檔案 + 行號

### 事實來源規則

| 資訊類型 | 事實來源 | 不應使用 |
|---|---|---|
| 頁面/元件名稱 | 原始碼常數或設定 | 翻譯、猜測 |
| API 路由 | 程式碼中的 Controller/Router 屬性 | 記憶、類似模式 |
| 類別/型別名稱 | 實際的類別定義 | API 文件（可能過時） |
| 資料結構 | 實際的 model/DAO 定義 | 回應範例 |
| 導覽/選單 | Layout/template 檔案 | 使用者描述 |

### 有疑問時
1. 先閱讀原始碼
2. 搜尋精確字串
3. 若仍不明確，詢問使用者
4. **絕不捏造資訊**

### 需求驗證
需求中每項事實性主張在實作前必須經過程式碼驗證：
- ✅ 檔案存在：驗證參考的檔案確實存在

- ✅ 資料來源：驗證頁面/元件實際呼叫了什麼（搜尋原始碼）
- ✅ 名稱：從原始碼常數讀取，絕不翻譯或猜測

> **黃金法則**：若無法為某事實性主張引用原始檔案與行號，應將其視為假設，而非事實。

## 實作完成報告（ICR）

### 目的
在多階段功能實作結束時建立的結構化產出物：
- **最終檢查點**：驗證所有程式碼已完成且建置乾淨
- **部署指南**：審查、測試及上線的檢查清單
- **交接文件**：為 QA、產品及維運團隊提供清晰的狀態
- **歷史記錄**：記錄決策、取捨及經驗教訓

### 何時建立
- ✅ 多階段實作（4 個以上檔案或多個層次）
- ✅ 有架構決策（選擇方案 A 而非 B/C）
- ✅ 建置驗證通過，0 個錯誤
- ✅ 功能開發結束（測試/Staging 前）

## 文件衛生（強制）

### 生產文件 vs. 工作文件

**生產文件**（持續使用，存放於 `.github/` 等已知位置）：
- 程式碼規範、架構知識、Agent/Skill 定義
- 僅在有明確意圖時修改 — 這些文件每天都被參考

**工作/分析文件**（臨時性，在對話過程中建立）：
- 調查筆記、規劃草稿、整合對應圖
- **規則**：存放在**一個**整理好的封存位置，不散落在工作區各處
- **命名**：使用日期 + 主題：`2025-12-10-consolidation-docs/`
- 絕不將臨時分析文件放在生產文件旁邊

### 規則
- ❌ 未收到明確請求時，不建立多份摘要/指南/參考文件
- ❌ 不將工作文件散落在 `.github/`、工作區根目錄或隨機資料夾
- ✅ 將生產文件保存在其指定資料夾
- ✅ 將工作文件封存在單一 `_archive/` 位置並使用清晰的命名
- ✅ 在建立新文件前詢問使用者
