# 文件共同編寫工作流程

此技能提供一個結構化的工作流程，引導使用者進行協作式文件建立。作為主動的引導者，帶領使用者完成三個階段：背景蒐集、精細化與結構化、讀者測試。

## 何時提供此工作流程

**觸發條件：**
- 使用者提到撰寫文件："write a doc"、"draft a proposal"、"create a spec"、"write up"
- 使用者提到特定文件類型："PRD"、"design doc"、"decision doc"、"RFC"
- 使用者似乎在開始一個重要的寫作任務

**初始提議：**
為使用者提供協作式文件編寫的結構化工作流程。說明三個階段：

1. **背景蒐集**：使用者提供所有相關背景資訊，Claude 提出澄清問題
2. **精細化與結構化**：透過腦力激盪與編輯，逐步建立各章節
3. **讀者測試**：使用全新的 Claude 實例（無對話上下文）測試文件，在他人閱讀前發現盲點

說明此方法有助於確保文件在他人閱讀時表現良好（包括當他們將其貼入 Claude）。詢問使用者是否想嘗試此工作流程或偏好自由寫作。

若使用者拒絕，以自由形式工作。若使用者同意，進入第 1 階段。

## 階段 1：背景蒐集

**目標**：縮小使用者所知與 Claude 所知的差距，以便後續提供聰慧的指導。

### 初始問題

先詢問使用者有關文件的中繼背景資訊：

1. 這是什麼類型的文件？（例如技術規格、決策文件、提案）
2. 主要讀者是誰？
3. 希望讀者在閱讀時產生什麼影響？
4. 是否有特定的範本或格式要遵循？
5. 還有其他約束或背景資訊需要知道嗎？

告知使用者可以簡短回答或以任何方式轉存資訊。

**若使用者提供範本或提及文件類型：**
- 詢問是否有範本文件可分享
- 若使用者提供共享文件的連結，使用適當的整合工具取得內容
- 若使用者提供檔案，讀取該檔案

**若使用者提及編輯現有共享文件：**
- 使用適當的整合工具讀取當前狀態
- 檢查是否有沒有替代文字的圖片
- 若圖片缺少替代文字，說明當他人使用 Claude 理解文件時，Claude 無法看到這些圖片。詢問是否需要生成替代文字。若需要，請求使用者將每個圖片貼入聊天以產生描述性的替代文字。

### 資訊轉存

初始問題回答後，鼓勵使用者轉存所有他們擁有的背景資訊。要求資訊如下：
- 專案/問題的背景
- 相關的團隊討論或共享文件
- 為什麼不使用其他解決方案
- 組織背景（團隊動態、過去事件、人事政治）
- 時間壓力或約束
- 技術架構或相依性
- 利益相關者的關注

建議使用者不要擔心組織——只是把所有資訊倒出來。提供多種提供背景的方式：
- 資訊流自由發言式
- 指向團隊頻道或執行緒
- 連結到共享文件

**若可用整合工具**（例如 Slack、Teams、Google Drive、SharePoint 或其他 MCP 伺服器），提及這些工具可用於直接拉入背景資訊。

**若無整合工具偵測到且在 Claude.ai 或 Claude 應用中**：建議使用者可在 Claude 設定中啟用連接器，以允許直接從訊息應用和文件儲存中拉入背景資訊。

告知使用者初始轉存後會提出澄清問題。

**背景蒐集期間：**

- 若使用者提及團隊頻道或共享文件：
  - 若可用整合工具：告知內容將被讀取，然後使用適當的整合工具
  - 若無可用整合工具：說明無法存取。建議在 Claude 設定中啟用連接器，或直接貼上相關內容。

- 若使用者提及不認識的實體/專案：
  - 詢問是否應該搜尋連接的工具以了解更多
  - 在搜尋前等待使用者確認

- 使用者提供背景資訊時，追蹤所學到的內容與仍不清楚的部分

**提出澄清問題：**

當使用者表示已完成初始轉存（或提供了大量背景資訊後），提出澄清問題以確保理解：

根據背景資訊的差距，產生 5-10 個編號問題。

告知使用者可以簡短回答（例如"1: 是，2: 見 #channel，3: 否因為向後相容"）、連結至更多文件、指向頻道閱讀或只是繼續轉存資訊。任何對他們最有效率的方式都可以。

**退出條件：**
當問題顯示理解時——當可以問到邊界情況與權衡而無需解釋基礎時，就已蒐集足夠背景資訊。

**過渡：**
詢問是否有其他背景資訊要在此階段提供，或是否可以開始草擬文件。

若使用者想添加更多，就讓他們。準備好時，進行第 2 階段。

## 階段 2：精細化與結構化

**目標**：透過腦力激盪、策劃與迭代精細化，逐章節建立文件。

**使用者指示：**
說明文件將逐章節建立。對每個章節：
1. 會提出有關應包含內容的澄清問題
2. 會腦力激盪 5-20 個選項
3. 使用者指出要保留/移除/合併的內容
4. 會草擬該章節
5. 透過手術式編輯進行精細化

從最多未知數的章節開始（通常是核心決策/提案），然後進行其餘部分。

**章節順序：**

若文件結構明確：
詢問使用者想從哪個章節開始。

建議從最多未知數的章節開始。對於決策文件，通常是核心提案；對於規格書，通常是技術方法。摘要章節最後做。

若使用者不知道需要什麼章節：
根據文件類型與範本，建議 3-5 個適合該文件類型的章節。

詢問此結構是否有效，或是否想調整。

**一旦同意結構：**

建立初始文件結構，所有章節均為佔位符文字。

**若可存取文件工具：**
使用 `create_file` 建立文件。這提供了 Claude 與使用者一個共同的腳手架。

告知將建立帶有所有章節標題和簡短佔位符文字（如「[待撰寫]」或「[此處內容]」）的初始結構。

建立帶有所有章節標題和佔位符文字的文件，提供檔案連結並指出現在是填充各章節的時候。

**若無法存取文件工具：**
在工作目錄中建立 Markdown 檔案。適當地命名（例如 `decision-doc.md`、`technical-spec.md`）。

告知將建立帶有所有章節標題和簡短佔位符文字的初始結構。

建立帶有所有章節標題和佔位符文字的檔案。

確認檔案名稱已建立並指出現在是填充各章節的時候。

**對每個章節：**

### 步驟 1：澄清問題

宣布將開始 [章節名稱] 的工作。提出 5-10 個有關應包含內容的澄清問題：

根據背景與章節目的產生 5-10 個具體問題。

告知使用者可以簡短回答或只是指出重要涵蓋的內容。

### 步驟 2：腦力激盪

對於 [章節名稱] 章節，腦力激盪可能包含的 [5-20] 項內容，取決於章節複雜性。尋找：
- 共享的背景資訊可能被遺忘的內容
- 尚未提及的角度或考慮

根據章節複雜性產生 5-20 個編號選項。最後，若想要更多選項，提出可進行進一步腦力激盪。

### 步驟 3：策劃

詢問哪些點應保留、移除或合併。要求簡短的理由以幫助了解未來章節的優先級。

提供範例：
- "保留 1,4,7,9"
- "移除 3（重複 1）"
- "移除 6（讀者已知）"
- "合併 11 和 12"

**若使用者給出自由形式的回饋**（例如「看起來不錯」或「我喜歡大部分，但是...」）而非編號選擇，擷取他們的偏好並繼續。解析他們想保留/移除/更改的內容並應用。

### 步驟 4：缺口檢查

根據他們選擇的內容，詢問 [章節名稱] 是否有任何重要遺漏。

### 步驟 5：草擬

使用 `str_replace` 將此章節的佔位符文字替換為實際草擬的內容。

宣布現在將根據他們選擇的內容草擬 [章節名稱] 章節。

**若使用文件工具：**
草擬後，提供文件的連結。

要求使用者閱讀並指出需要改變的內容。說明具體有助於了解後續章節的風格。

**若使用檔案（無文件工具）：**
草擬後，確認完成。

告知 [章節名稱] 已在 [檔案名稱] 中草擬。要求使用者閱讀並指出需要改變的內容。說明具體有助於了解後續章節的風格。

**使用者的關鍵指示（在草擬第一個章節時包含）：**
提供備註：與其直接編輯文件，請表示想改變什麼。這有助於了解他們的寫作風格。例如："移除 X 要點——已被 Y 涵蓋"或"讓第三段更簡潔"。

### 步驟 6：迭代精細化

使用者提供回饋時：
- 使用 `str_replace` 進行編輯（絕不重新列印整個文件）
- **若使用文件工具**：每次編輯後提供文件連結
- **若使用檔案**：只需確認編輯完成
- 若使用者直接編輯文件並要求閱讀：心理上記錄他們所做的更改並牢記（這顯示他們的偏好）

**持續迭代**直到使用者對章節滿意。

### 品質檢查

連續 3 次迭代無實質性更改後，詢問是否有任何可移除而不失去重要資訊的內容。

章節完成時，確認 [章節名稱] 完成。詢問是否準備好進行下一個章節。

**對所有章節重複。**

### 接近完成

當接近完成（80%+ 章節已完成）時，宣布重新閱讀整個文件的意圖並檢查：
- 各章節之間的流暢性與一致性
- 冗餘或矛盾
- 任何感覺像「廢話」或通用填充物的內容
- 每句話是否都有份量

閱讀整個文件並提供回饋。

**當所有章節都已草擬與精細化：**
宣布所有章節已草擬。指出重新審視完整文件的意圖。

檢查整體連貫性、流暢性、完整性。

提供任何最終建議。

詢問是否準備好進入讀者測試，或是否想精細化其他內容。

## 階段 3：讀者測試

**目標**：使用全新的 Claude 實例（無上下文混淆）測試文件，驗證其對讀者是否有效。

**使用者指示：**
說明現在將進行測試以查看文件是否真的對讀者有效。這捕捉盲點——對作者有意義但可能讓他人困惑的事項。

### 測試方法

**若可存取子代理**（例如在 Claude Code 中）：

直接進行測試，無需使用者參與。

### 步驟 1：預測讀者問題

宣布預測讀者在嘗試了解此文件時可能提出的問題。

產生 5-10 個讀者可能實際提出的問題。

### 步驟 2：使用子代理測試

宣布這些問題將使用全新的 Claude 實例（無此對話的上下文）進行測試。

對每個問題，使用僅包含文件內容與問題的子代理。

總結讀者 Claude 對每個問題的正確/錯誤表現。

### 步驟 3：執行額外檢查

宣布將執行額外檢查。

呼叫子代理檢查歧義、虛假假設、矛盾。

總結找到的任何問題。

### 步驟 4：報告與修正

若找到問題：
報告讀者 Claude 在特定問題上遇到困難。

列出具體問題。

指出修復這些缺口的意圖。

循環回精細化階段以解決有問題的章節。

---

**若無子代理存取**（例如 claude.ai 網頁界面）：

使用者需要手動進行測試。

### 步驟 1：預測讀者問題

詢問人們在嘗試了解此文件時可能提出什麼問題。他們會在 Claude.ai 中輸入什麼？

產生 5-10 個讀者可能實際提出的問題。

### 步驟 2：設定測試

提供測試指示：
1. 開啟全新的 Claude 對話：https://claude.ai
2. 貼上或分享文件內容（若使用啟用連接器的共享文件平台，提供連結）
3. 詢問讀者 Claude 產生的問題

對每個問題，指示讀者 Claude 提供：
- 答案
- 是否有任何模糊或不清楚的地方
- 文件假設讀者已知道什麼知識/背景

檢查讀者 Claude 是否提供正確答案或誤解任何內容。

### 步驟 3：額外檢查

同時詢問讀者 Claude：
- "此文件中的什麼對讀者可能模糊或不清楚？"
- "此文件假設讀者已知道什麼知識或背景？"
- "是否有任何內部矛盾或不一致？"

### 步驟 4：根據結果迭代

詢問讀者 Claude 做錯了什麼或遇到困難的地方。指出修復那些缺口的意圖。

循環回精細化階段以解決任何有問題的章節。

---

### 退出條件（兩種方法）

當讀者 Claude 持續正確回答問題並且不再提出新的缺口或歧義時，文件已準備好。

## 最終審查

當讀者測試通過時：
宣布文件已通過讀者 Claude 測試。完成前：

1. 建議他們自己進行最終閱讀——他們擁有此文件並對其品質負責
2. 建議檢查任何事實、連結或技術細節
3. 要求驗證是否達到他們想要的影響

詢問是否想進行一次最後審查，或工作是否完成。

**若使用者想最後審查，提供。否則：**
宣布文件完成。提供一些最終提示：
- 考慮在附錄中連結此對話，讓讀者看到文件是如何開發的
- 使用附錄提供深度而不會讓主文件膨脹
- 隨著真實讀者回饋的接收而更新文件

## 有效指導的提示

**語氣：**
- 直接且程序性
- 簡要說明影響使用者行為的理由
- 不試圖「推銷」方法——只是執行它

**處理偏差：**
- 若使用者想跳過一個階段：詢問是否想跳過這個且自由寫作
- 若使用者似乎沮喪：承認這比預期花費更長時間。建議加快速度的方法
- 始終給予使用者調整流程的權力

**背景管理：**
- 始終，若提及某事物的背景缺失，主動詢問
- 不要讓缺口累積——隨著它們出現處理

**文件管理：**
- 使用 `create_file` 草擬完整章節
- 對所有編輯使用 `str_replace`
- 每次更改後提供文件連結
- 絕不使用文件進行腦力激盪清單——那只是對話

**品質優於速度：**
- 不要急於完成階段
- 每次迭代應進行有意義的改進
- 目標是一個真正對讀者有效的文件
