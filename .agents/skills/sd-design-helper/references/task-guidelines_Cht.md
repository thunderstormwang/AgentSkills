# 任務階段指導方針

這些方針嚴格適用於 `sd-design-helper` 開發生命週期的 **第 4 階段 — Task**。

---

## 任務限制 (Task Constraints)

- **檔案限制**：每個任務修改的檔案數**不得超過 3 個**。
- **DB Schema 變更**：若 Design 包含 DB 變動，必須建立一個專門的任務來產出 SQL 腳本。
    - **儲存路徑**：腳本必須儲存在專案根目錄的 `sql/` 資料夾中。
    - **檔名格式**：`PXBOX-{jira ticket no}.sql`。
    - **Jira 編號**：若 Jira ticket 編號不明，Agent 必須向使用者確認。
    - **執行方式**：註明該腳本將由人工在各個環境中手動執行。
- **API 合約變更**：若 Design 包含 API Request/Response 的變動，必須建立一個專門的任務來產出「前端 API 變動說明」，以利前端人員從 Swagger 查找。
    - **內容要求**：說明必須包含 API 路由、變動類型（新增/修改/刪除 API），以及具體的欄位變動細節（新增/修改/刪除 Request/Response 中的欄位）。
    - **交付方式**：此說明應直接放在該 Task 的描述中，以便使用者複製貼上。
- **任務進程**：始終遵循「功能優先，優化其次」。

---

## 內容要求 (Content Requirements)

每個任務的描述必須足夠詳細，使開發者無需反覆查閱 Design 即可實作。必須包含：

- **引用 (Reference)**：該任務實作的 Design ID (例如：`[Ref: D1]`)。
- **目標專案**：專案名稱或 Assembly 名稱。
- **組件 (Component)**：具體的類別名稱（例如：Handler, Controller, Service）。
- **方法 (Methods)**：要新增或修改的方法名稱。
- **邏輯細節**：逐步的邏輯說明、程式碼模式、或具體的驗證規則。
- **單元測試**：每個涉及邏輯變動的任務，都必須透過單元測試進行驗證。單元測試與生產程式碼的修改可以在同一個任務中，也可以拆分成不同的任務（例如：T1 寫生產程式碼，T2 寫單元測試），以維持 3 個檔案的限制。

---

## 格式範例 (Format Example)

使用結構化列表而非表格，以提高可讀性：

### T1: [任務名稱]
- **引用**：`[D1]`
- **目標**：`[專案名稱]` -> `[類別名稱]` -> `[方法名稱]`
- **實作細節**：
    - [步驟 1：具體邏輯/指令]
    - [步驟 2：具體邏輯/指令]
    - [單元測試：描述要增加/更新的測試案例]
- **影響檔案**：(最多列出 3 個檔案)
