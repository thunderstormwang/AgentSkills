# 任務階段指導方針

這些方針嚴格適用於 `sd-design-helper` 開發生命週期的 **第 4 階段 — Task**。核心目標是產出高品質、可獨立執行的任務清單。

---

## 任務結構與限制 (Task Structure & Constraints)

- **邏輯 Commit 粒度**：每個任務應理想地對應一個邏輯 commit。
- **檔案限制**：每個任務建議修改的檔案數**不得超過 3 個**，以確保清晰度與可維護性。
- **DB Schema 變更**：若涉及 DB 變動，必須建立一個產出 SQL 腳本的任務。
    - **儲存路徑**：腳本必須儲存在專案根目錄的 `sql/` 資料夾中。
    - **檔名格式**：`PXBOX-{jira ticket no}.sql`。
    - **Jira 編號**：若編號不明，必須向使用者確認。
- **API 合約變更**：若涉及 API 變動，必須建立一個產出「前端 API 變動說明」的任務。
    - **內容要求**：包含 API 路由、變動類型（新增/修改/刪除），以及具體的欄位變動細節。

---

## 任務編排順序建議 (Task Ordering)

為了確保前後端並行開發與整合效率，在產出 Task 清單時應遵循以下順序：

1.  **DB Schema 變更**：優先產出 SQL 腳本，這是所有變更的基礎。
2.  **Entity / Domain 變更**：定義核心業務邏輯與資料結構。
3.  **API 空殼與欄位**：優先定義 API Request/Response 模型與 Controller 端點（允許先使用佔位邏輯）。
4.  **API 摘要**：在 API 合約定義完成後立即提供前端摘要，以利合作開發。
5.  **詳細功能實作**：實作具體業務邏輯與優化。

---

## 內容要求 (Content Requirements)

每個任務的描述必須足夠詳細，使後續執行者無需回頭參考 Design 即可實作。必須包含：

- **引用 (Reference)**：該任務實作的 Design ID (例如：`[Ref: D1]`)。
- **目標專案**：專案名稱或組件名稱。
- **組件 (Component)**：具體的類別名稱（例如：Handler, Controller, Service）。
- **方法 (Methods)**：要新增或修改的方法名稱。
- **邏輯細節**：逐步的邏輯說明、程式碼模式、或具體的驗證規則。
- **單元測試**：描述要增加或更新的測試案例。

---

## 格式範例 (Format Example)

### T1: [任務名稱]
- **引用**：`[D1]`
- **目標**：`[專案名稱]` -> `[類別名稱]` -> `[方法名稱]`
- **實作細節**：
    - [步驟 1：具體邏輯/指令]
    - [步驟 2：具體邏輯/指令]
    - [單元測試：描述要增加/更新的測試案例]
- **影響檔案**：(最多列出 3 個檔案)
