# 20260316 PXBox-26379 行銷檔期重覆檢查優化  

## 會議記錄

現狀為同類型行銷活動的檔期不可重疊:
- 首購行銷活動不可與其它首購行銷活動的檔期重疊
- 第N件折行銷活動不可與其它第N件折行銷活動的檔期重疊

預期改動為單一商品在特定互斥組別內，檔期不可重疊:  
- 訂單滿件 - 則不能與其它訂單滿件和第N件折的檔期重疊  
- 訂單滿額 - 則不能與其它訂單滿額和第N件折的檔期重疊  
- 首購 - 則不能與其它首購的檔期重疊  
- 第N件折 - 則不能與其它第N件折、訂單滿額和訂單滿件的檔期重疊  

## 儲存體評估

估算資料量
- 100 萬個商品  
- 每個商品被掛上的行銷檔期平均有 25 個  
- 一次行銷異動最多可異動 30 萬個商品 和 10 個檔期  

等於每次儲存要進行 2,500 萬筆基礎資料 vs. 300 萬筆異動資料 的檔期重覆檢查  

### Redis

zset:  
- member: {promotion_id}.{starttime 的 timestamp}.{endtime 的 timestamp}
- value: {endtime 的 timestamp}

用 zset 的 ZRANGEBYSCORE 拉出所有「在目標檔期的結束時間之前就已經開始」的檔期，再拆解 member 確定是否重疊。  

如果估算 zset 中一組 member 加 value 會佔 40 byte，則  
redis 須常駐 100萬 * 25 * 40 byte = 100,000 萬 byte = 1GB  
一次最多送給 redis 30萬 * 10 * 40 byte = 120,000 萬 byte = 120MB  

在 reids 的常駐空間太高  
在大量比對和大量更新時，須將指令批次送進 redis  
因為需求有將行銷活動分組，實際上一個商品在 redis 會需要多個 zset  
為了避免從 redis 拿回大量的資枓，須撰寫 lua 腳本將部份計算留在 redis 端，維護麻煩  

### Clickhouse

不同於傳統 OLTP 資料庫(如 MySQL)的行式架構，ClickHouse 採用「列式儲存」：
- 極速 I/O (Sequential I/O)： 相同欄位（如 ProductId, StartTime）在磁碟中是連續存放的。在進行大規模檔期檢查時，系統是順序讀取 (Sequential I/O) 必要的欄位，而非像 OLTP 必須在整張表內隨機存取 (Random I/O)，大幅降低磁碟尋址延遲。  
- 向量化運算 (SIMD)： 由於同欄位資料在記憶體中排列緊湊且型別一致，ClickHouse 能調用 CPU 的 SIMD (單指令多資料流) 指令集，在一個時脈週期內同時比對多組數據。這將運算從「逐行判斷」提升至「向量化處理」，效能有量級上的突破。  

Table 欄位至少有 product_id, promotion_id, start_time, end_time, 假設估計是 40 byte  
在硬碟要佔掉 100萬 * 25 * 40 byte = 100,000 萬 byte = 1GB  
一次最多送給 clickhouse 30萬 * 10 * 40 byte = 120,000 萬 byte = 120MB  

#### 資料傳輸階段 (C# → ClickHouse)  
數據量： 120 MB  
傳輸環境： 假設為標準內網 1 Gbps(約 125 MB/s）。  
預估時間： 約 1 ~ 1.5 秒。

#### 建立 Hash Table 階段((Build Phase)
動作： ClickHouse 將這 300 萬筆新資料在記憶體中建立「高速字典（Multimap）」。  
效能基準： ClickHouse 在現代伺服器 CPU 上處理 Hash 運算的速度大約是每秒一千萬列。  
預估時間： 約 0.3 ~ 0.5 秒。  

#### 向量化比對階段 (Probe & Overlap Check)  
動作： 掃描正式表 2,500 萬筆資料，並進行 (NewStart < ExistEnd) AND (NewEnd > ExistStart) 的比對。  
處理邏輯：
- I/O 讀取： 雖然總量 1GB，但比對時只讀取需要的欄位。這意味著磁碟只需要順序讀取約 200 MB 的數據。  
- 向量運算： 利用 SIMD 指令，一組 CPU 核心可以每秒處理數億次的邏輯比對。  

預估時間： 約 0.5 ~ 1.5 秒。

### 後端 - Coupon - 提供可呼叫的檔期重覆邏輯檢查的接口  

接口為方法，須定義方法的參數  
須考慮可能不會被呼叫  
要符合需求描述的能去檢查其它行銷活動的檔期是否重覆  
實作 C# 連接 clickhouse 的底層  

範例 sql
```sql
-- 建議建立一個暫存表來存放這次要檢查的 30 萬筆資料
CREATE TEMPORARY TABLE promotion_schedules_check AS promotion_schedules_formal;

SELECT 
    new.ProductId,
    new.ScheduleId AS New_Schedule,
    old.ScheduleId AS Conflict_Schedule,
    new.StartTime,
    new.EndTime,
    old.StartTime AS Confict_Start,
    old.EndTime AS Conflict_End
FROM marketing_schedules_check AS new
-- 直接與正式表進行 JOIN
INNER JOIN marketing_schedules_formal AS old 
    ON new.ProductId = old.ProductId
WHERE 
    -- 核心比對邏輯：(A.Start < B.End) AND (A.End > B.Start)
    (new.StartTime < old.EndTime) AND (new.EndTime > old.StartTime)
-- 設定：確保 JOIN 使用效能最佳化模式
SETTINGS join_use_nulls = 1;
```

### 後端 - Coupon - 提供可呼叫的更新商品 X 行銷檔期資料的接口  

接口為方法，須定義方法的參數  
實作 C# 連接 clickhouse 的底層  

#### 方案一: ReplacingMergeTree(增量覆蓋模式)  

用 `ReplacingMergeTree(UpdateTime)` 的機制，用新版本的資料覆蓋掉舊版本  
商品被刪除也要新增資料，將 `IsDelete` 標記刪除  

優點： 寫入效能極高，對 C# 來說邏輯最簡單(通通都是 Insert)。  
缺點： 查詢時加上 FINAL 會損耗效能。  

範例 sql
```sql
SELECT * FROM marketing_schedules_formal FINAL WHERE IsDeleted = 0;
```

#### 方案二: Exchange Tables(全量/影子切換模式)  

建立影子表  
重倒資料  
原子切換  

```sql
INSERT INTO marketing_schedules_new
SELECT * FROM (
    -- A: 抓出沒被這次異動影響到的舊資料
    SELECT * FROM marketing_schedules_formal
    WHERE ProductId NOT IN (SELECT DISTINCT ProductId FROM temp_check_table)

    UNION ALL

    -- B: 抓出這次異動後的新資料 (包含新增與修改，不包含移除的)
    SELECT * FROM temp_check_table WHERE Action != 'Delete'
)
```

```sql
EXCHANGE TABLES marketing_schedules_formal AND marketing_schedules_new;
```

優點： 查詢效能最強(跟一般 Table 一樣)，且切換是原子性的。  
缺點： 同步時需要重組資料，會產生額外的磁碟 I/O。但在 100 萬筆的規模下，這個動作通常在 5-10 秒內就能完成。
