# 商品庫存遷移至 Redis 評估文件

為了應對高併發情境並減少資料庫壓力，本文件規劃將『商品庫存』數據從關聯式資料庫 (RDBMS) 遷移至 Redis，並確保資料的一致性與回寫機制。

## 1. Req (需求分析)

- **Objective (目標):** 
  - 降低高併發下主資料庫的 I/O 負擔。
  - 提升庫存查詢與扣減的反應速度 (毫秒級)。
  - 確保庫存不超賣，並能正確同步回資料庫。
- **Current State (當前狀態):** 
  - 商品庫存存放於 SQL 資料庫的 `ProductStock` 資料表。
  - 每次下單皆會觸發 `UPDATE` 語句並鎖定行紀錄，高流量時會導致大量庫存更新請求排隊。
- **Proposed Changes (計畫變更):** 
  - 引入 Redis 作為庫存的緩存與即時扣減引擎。
  - 使用 Redis Lua 腳本確保庫存扣減的原子性 (Atomicity)。
  - 建立非同步回寫機制，將 Redis 的變動更新至資料庫。
- **Conclusions (結論):** 
  - 以 Redis 作為「庫存即時狀態」，DB 作為「庫存最終歸檔」。
  - 庫存初始化時從 DB 加載至 Redis。
- **Constraints (限制):** 
  - Redis 需配置 RDB/AOF 以防止宕機導致數據完全遺失。
  - 需要考慮 Redis 與 DB 之間的最終一致性延遲。

## 2. Design (技術設計)

### Technical Decisions (ADR)
- **Decision:** 使用 Redis Lua 腳本執行庫存扣減。
- **Context:** 在高併發下，`GET` 後再 `SET` 會導致 Race Condition。
- **Consequences:** 確保庫存檢查與扣減在單一原子操作內完成，避免超賣。

### Detailed Design Table
| 組件 | 變更類型 | 詳細說明 (Schema, Logic, Cache) |
| :--- | :--- | :--- |
| **Redis** | **Cache** | Key: `inventory:{sku_id}`, Type: `String` (存儲庫存數量) |
| **InventoryService** | **Logic** | 實作 `StockDeduct` 方法，調用 Lua 腳本：`if redis.call('get', KEYS[1]) >= ARGV[1] then return redis.call('decrby', KEYS[1], ARGV[1]) else return -1 end` |
| **SyncWorker** | **Job/MQ** | 監聽 Redis 變動或定期掃描，將最新庫存值寫回資料庫 `ProductStock` 資料表。 |
